# v0.2.24 更新说明

## 重点改动：1688 收货地址同步与失效拦截

### 问题表现

部分采购单点「推送 1688 下单」时弹出：

`1688 收货地址 ID 无效，请到「询盘设置」点「同步 1688 地址」后重新选择地址再推单。`

或者更早期版本里看到 1688 原始报错：

`errorCode:AddressId invalid`

界面上推单按钮按了没反应，但用户找不到「同步 1688 地址」按钮在哪。

### 根因

1688 开放平台**没有提供「创建收货地址」的 API**，第三方应用只能通过 `alibaba.trade.receiveAddress.get` 拉买家在 1688 后台已经维护好的地址。我们这边的本地表 `erp_1688_delivery_addresses` 跟远端容易出现三类不一致：

1. 远端这条地址被买家在 1688 后台删掉了，本地却还留着 → 推单时 1688 校验失败
2. 多 1688 买家账号场景下，本地存的 `address_id` 实际属于另一个买家账号 → 跨账号无效
3. 本地从来没同步过 / 同步过的数据缺 `address_id` → 推单直接报缺字段

之前同步逻辑只增不删，本地的"幽灵地址"一直堆着；UI 上也没有显式的「同步 1688 地址」入口，只有打开「店铺管理 → 新增店铺账号」弹窗时本地为空才会自动隐式触发一次。

### 修复内容

1. **同步逻辑改为 diff 模式**：拉回远端列表后，本地有但远端没有的行直接打 `inactive`，不再参与推单选项；接口返回新增 / 更新 / 失效三栏计数。安全策略：远端返回为空时跳过反向标记，避免一次接口失败误伤所有本地地址。
2. **推单前本地预校验**：地址 `status` 不是 `active` 或 `address_id` 缺失，直接抛 `ADDRESS_INACTIVE` / `ADDRESS_REMOTE_ID_MISSING`，省掉一次无谓的 1688 网络调用，错误也更精准。
3. **询盘设置加显式入口**：在「采购中心 → 询盘设置」面板底部新增 1688 收货地址区块，含两个按钮：
   - 「同步 1688 地址」：调 `alibaba.trade.receiveAddress.get`，toast 反馈「新增 X / 更新 Y / 失效 Z（共 N 条）」
   - 「打开 1688 后台」：跳到 `https://work.1688.com/`，让用户在那边新建/修改地址后回来再点同步
4. **错误提示自动恢复**：推单 / 预览撞到 `AddressId invalid` / `ADDRESS_ID_INVALID` / `ADDRESS_INACTIVE` / `ADDRESS_REMOTE_ID_MISSING` 任一种，前端会**后台静默触发一次同步**并提示用户「已自动同步：N 条失效地址被清理，请重新选择地址再推单」，不再让用户自己摸索。

### 用户操作流程（卡住时怎么办）

1. 推单报错弹「1688 收货地址 ID 无效…」 → 不用做任何操作，系统已经自动同步过了
2. 重新点「推送 1688 下单」，在弹出的地址选择 Modal 里挑一条仍然有效的地址即可
3. 如果买家账号下根本没有可用地址：
   - 点采购中心顶部「询盘设置」
   - 点底部「打开 1688 后台」按钮，进入 1688 web 后台维护好新地址
   - 回到询盘设置点「同步 1688 地址」
   - 重新推单

### 不会做的事

- **在 ERP 里直接创建 1688 收货地址**：1688 开放平台从未对第三方应用开放写地址的接口（trade 类目下只有 `receiveAddress.get`），同样的限制对所有 ERP 服务商生效，不是本系统的设计偏好。
- **推单失败自动用别条地址重推**：可能把潜在错单变成隐性错单，更危险，所以一律由用户重新选择。

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」，下载完成后重启客户端。
