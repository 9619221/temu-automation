# v0.2.18 更新说明

## 重点改动：以图搜款加 Playwright 真 Chrome 兜底，绕过 1688 mtop 反爬

### 0.2.17 还没解决的根因
v0.2.17 的网络诊断按钮已经定位到：**坏电脑的家用宽带 IP 被 1688 反爬列入黑名单**（`rgv587_flag: sm` + `deny_h5/punish`）。这跟主控端云 IP 当初一模一样的症状。

只要这台电脑去裸 fetch `search.1688.com`，1688 就直接 deny。换网络 / 等几小时只是临时绕开，不是根因解。

### 0.2.18 真正的根因解
项目里其实已经有一条「绕过 mtop 反爬」的路：`fetch1688CostByImage` 用 Playwright 真 Chrome（带用户登录态 cookies + 真实 fingerprint）打开 1688 自家的图搜页 `https://air.1688.com/app/channel-fe/search/index.html#/result?image_list=...`。但之前只用来抓价格中位数，没用来做候选商品导入。

这版做了两件事：

1. **新加 `search1688OffersByImage`**：扩展 `aliexpress-1688-cost.mjs`，在同一条 Playwright 流水线上把 1688 air 图搜页的完整 offer 列表（offerId/title/imageUrl/productUrl/price/supplier/sales）都抓出来。优先 XHR 拦截（结构稳），DOM 选择器兜底。

2. **`buildClientImageSearchMockResults` 三级 fallback 链**：
   - **第一级**：直传 1688 mtop（家用 IP 没被标黑时，5-10s 拿候选）
   - **第二级**（新增）：mtop 失败 / 0 候选时，调用 `worker.mjs` 的 `search_1688_image` action，用 Playwright 真 Chrome 走 1688 air 图搜（30-45s，但绕开反爬）
   - **第三级**：再失败就让主控端跑 alphashop（永远 0 候选但秒回，至少不卡死）

### 为什么 Playwright 真 Chrome 能过反爬
`rgv587_flag` / `cloud_ip_bl` 这些反爬命中的判断综合：
- 请求 cookies 是否带 `_m_h5_tk`、`cna`、登录态 `_nk_` 等
- User-Agent / fingerprint 是否真实
- 行为模式（鼠标/滚动/页面停留）
- IP 信誉

裸 fetch 走 mtop **全 0 分** ——没有完整 cookies，没有 fingerprint，没有行为，IP 还可能被标黑。
真 Chrome + 用户已登录的 1688 profile **几乎 100 分**——跟正常用户访问一模一样。

即使**真 Chrome 也被弹滑块**了（极端情况），用户能在已经打开的浏览器窗口里手动过一下，下一次就通了。我们的代码会捕捉这个状态并提示用户。

### 触发条件
完全自动：你点「以图搜款」之后，如果 mtop 在 30s 内成功（家用 IP 干净）就直接出结果；如果 mtop 0 候选或挂了，自动 fallback 到 Playwright，**不需要手动切换**。

### 已知限制
- Playwright 路径需要 1688 已经登录（用工具的「1688 登录」按钮过一次就行，cookies 持久化在 profile 里）
- 主控端（Linux 服务器，没 GUI 跑不了 Chrome）没 workerInvoker，所以不会启用第二级 fallback，依旧只能走 mtop（如果连云 IP 黑名单也没解决）→ alphashop（永远 0）。这条链路真要修，得申请 1688 官方 ACL（长期方案，需要在 1688 开放平台后台申请）。

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」点一下，关闭工具再重新打开。两台都升。
