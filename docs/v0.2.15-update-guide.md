# v0.2.15 更新说明

## 重点改动：1688 web 图搜结果不再带 raw → 客户端上传从 200KB 变 5KB

### 0.2.14 仍有的问题
- 「以图搜款」在两台电脑测，一台秒成功，一台撞 `IPC erp:purchase:action 调用超时 (120s)`
- 主控端日志看 `[handlePurchaseAction t=18685ms] payload read` —— 客户端**上传 payload 就花了 18 秒**
- 网络好的能熬过去（4-18s），网络差的（慢上行 ADSL / 4G / 拥塞 wifi）撑不到 120s

### 实测拆解出的根因
v0.2.13 把客户端 1688 web mtop 图搜默认开启之后，每条搜款结果（`normalizeWebOffer` 输出）会带一个 `raw` 字段，里头是完整的 1688 mtop offer 对象（aliTalk / brand / commonPositionLabels / marketOfferTag / trackInfoModel / tradeService / midComplexLabels …），单条 14-21KB。

10 条候选 → 上传 payload 就有 **200KB+**。慢上行的客户上传时间：

| 上行速度 | 200KB 上传 |
|---|---|
| 50Mbps 光纤 | ~30ms |
| 5Mbps 光纤 | ~320ms |
| 1Mbps ADSL | ~2s |
| 200Kbps 慢 ADSL / 拥塞 4G | ~10s |
| 100Kbps 极慢 | **20s+** |

加上 TLS 握手反复、丢包重传，慢的客户能撞 120s。

UI 和主控端**都不用** raw 这堆字段，UI 只读顶层 `productTitle` / `imageUrl` / `productUrl` / `soldOut` / `unitPrice` / `supplierName`，那些 `normalizeWebOffer` 已经抽到顶层了。

### 0.2.15 修复
1. **`normalizeWebOffer` 不再回 raw**：每条 offer 输出从 ~15KB 降到 ~500 字节
2. 把销量数字（`tradeQuantity.tradeQuantity` 等）抽到顶层 `soldOut` / `sales`，UI 销量徽标继续能显示
3. 主控端审计需要原始 1688 数据时，可以通过 `productUrl` 反查 detail.1688.com

### 效果
- 客户端上传 payload：**200KB → 5KB**（缩小 40 倍）
- 慢上行用户上传时间：18s → <500ms

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」点一下。两台都要升。

## 服务器侧

主控端 `erp.temu.chat` 同步部署，无需额外操作。
