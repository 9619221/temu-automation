# v0.2.13 更新说明

## 重点改动：以图搜款默认走客户端宽带 IP，绕过主控端云 IP 反爬黑名单

### 0.2.12 仍有的问题
- 「以图搜款」点完弹「这张图没有搜到候选」，alphashop 返回的 `result: []`
- 不是网络/超时问题，是搜款链路三层全断了

### 实测拆解出的根因
1. **alphashop `imageSearchProduct`**：传入 `https://erp.temu.chat/...` 这种非阿里 CDN 的 imgUrl 时静默返回空数组（HTTP 200 / resultCode SUCCESS / result []）。换成 `https://cbu01.alicdn.com/...` 立刻能搜到 10 个商品 —— 它根本没去下载我们的图，只对已经在阿里 CDN 上的图做特征匹配。
2. **1688 官方 `product.image.upload`**：当前 AppKey 没 ACL（之前日志已记录）。
3. **1688 网页版 mtop 搜款**：从主控端云服务器调用直接被反爬封，返回 `{ rgv587_flag: "sm", url: ".../deny_h5.html?...|cloud_ip_bl|0..." }`。`cloud_ip_bl` = cloud IP blacklist。

也就是说**用户上传的非阿里 CDN 图，从主控端那边搜款是必死的**。唯一活路是从用户家里宽带的桌面客户端直接打 1688 mtop —— 而这条路被 v0.2.11 默认关掉了（当时为了省 IPC 60s 超时窗口）。

### 0.2.13 修复
1. **`buildClientImageSearchMockResults` 默认开启**：客户端模式下「以图搜款」会先在本机调 1688 mtop（putImage + searchOffer），把结果作为 `mockResults` 一起带给主控端，绕过 alphashop URL 限制和云 IP 反爬黑名单。
2. **timeout 从 25s 提到 30s**：mtop putImage 实测 6-12s，留点 buffer。
3. 想强制关闭可以设 `ERP_CLIENT_IMAGE_PRESEARCH=0`。

### 验证
- 从主控端调 1688 mtop：返回 `cloud_ip_bl` deny ❌
- 从客户端 Windows 调 1688 mtop：8.3s 返回 10 个商品 ✅（实测拿用户那张洗车刷图，第一条结果是「厂家直供静电除尘掸…纤维掸家用扫灰掸」，明显匹中了）

### 长期改进方向（不在本版处理）
- 申请 1688 开放平台 `product.image.upload` 接口的 ACL（用户在 1688 开放平台后台操作）
- 拿到 ACL 后主控端搜款能直接走官方接口，不再依赖客户端宽带 IP

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」点一下。

## 服务器侧

主控端 `erp.temu.chat` 同步部署，无需额外操作。
