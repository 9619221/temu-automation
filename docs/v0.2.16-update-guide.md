# v0.2.16 更新说明

## 重点改动：客户端 1688 mtop 预搜加 30s 硬上限，超时自动 fallback 主控端

### 0.2.15 仍有的问题
- 同样的代码、同样的服务器、同样的 1688 账号，**两台用户电脑一台秒成功一台撞 120s 超时**
- 服务器日志看：成功的那次 `payload read t=0ms`，失败的那次根本看不到请求到达
- 浏览器测 `https://erp.temu.chat/health` 又通的

也就是说请求**卡在客户端本机的 1688 mtop 预搜阶段**，根本没发到主控端。

### 实测拆解出的根因
v0.2.13 把客户端 1688 mtop 图搜默认开启之后，每次「以图搜款」都要从用户家里 IP 打 4 次 1688 接口：

1. `warmCnaCookie` （mmstat.com） —— 上限 8s
2. `acquireMtopToken` （h5api.m.1688.com） —— 上限 30s
3. `uploadImage` （h5api.m.1688.com 传图） —— 上限 30s
4. `searchOffers` （search.1688.com） —— 上限 30s

理论最坏 **98 秒**。再加上后续 remoteRequest 走 caddy 到主控端、IPC 主→渲染回传，加起来很容易冲过预设的 120s IPC 软超时。

具体出问题的那台电脑大概率是：
- 偶尔被 1688 mtop 列入临时反爬名单（`rgv587_flag: sm`）
- 公司/家用路由对国外/外网 TCP idle 比较狠
- IPv4 → IPv6 切换、CDN 节点选优问题

任何一个组合都可能让某一步 hang 在 socket 层，AbortController 在那种状态有时候捅不动。

### 0.2.16 修复
**给 `buildClientImageSearchMockResults` 套了一个 30s 硬上限**。每一步包一层 Promise.race，任何阶段（fetch image / token / putImage / searchOffer）累加超过 30s 直接放弃，原 payload 透传给主控端，让 alphashop 兜底。

效果：

| 情况 | 之前 | 现在 |
|---|---|---|
| 客户端 mtop 顺畅 | 5-10s 拿候选 | 5-10s 拿候选（不变）|
| 客户端 mtop 慢 / 偶发 deny | **撞 IPC 120s 超时** | 30s 内放弃，fallback 主控端 alphashop，提示「这张图没有搜到候选」 |
| 客户端 mtop 完全挂 | 撞 IPC 120s 超时 | 30s 内放弃，主控端 fallback |

也就是说**最差情况下**用户会在 30 秒后看到「这张图没有搜到候选，可以换一张更清晰的商品主图再试」（alphashop 对外站图永远 0 个），**而不是干等 120 秒看一个 IPC 超时弹窗**。

### 长期改进方向（不在本版处理）
- 申请 1688 开放平台 `product.image.upload` 接口的 ACL，主控端搜款不再依赖客户端 IP
- 或者：在 Electron renderer 里开一个隐藏 Webview 走真浏览器（携带 1688 标准登录态）做图搜，比裸 fetch 走 mtop 稳

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」点一下，关闭工具再重新打开。两台都升。
