# v0.2.19 更新说明

## 重点改动：修复用户端以图搜款 120s IPC 超时

### 问题表现

用户端在采购中心点「以图搜款」后，偶发弹出：

`IPC erp:purchase:action 调用超时 (120s)`

界面里采购单会停在「已找货源」，但候选商品仍是 0 个，用户看不到真正原因。

### 根因

客户端模式下，以图搜款会先在用户本机跑 1688 预搜链路：

1. 裸 `1688 mtop` 图搜，最多约 30s。
2. mtop 失败或 0 候选后，再用 Playwright 真 Chrome 打开 1688 air 图搜，最多约 60s。
3. 如果还失败，再把请求交给主控端 `/api/purchase/action`。

这三段串行执行，但前端 IPC 软超时只有 120s。遇到 1688 反爬、滑块、未登录、网络慢时，本机预搜先耗掉大部分时间，主控端再处理 workbench 返回时就容易撞 120s。

同时，之前代码会吞掉「需要登录 / 需要过滑块」这类可处理错误，最终只显示 IPC 超时。

### 修复内容

1. 客户端预搜总预算固定为 75s：
   - mtop 最多 30s
   - Playwright 真 Chrome 最多 45s
2. Playwright 检测到 1688 登录、滑块、人机验证时，直接把错误透传给前端。
3. 本机 mtop + 真 Chrome 都没候选时，返回空 `mockResults` 给主控端，只做状态和 workbench 返回，不再让主控端重复跑一轮慢图搜。
4. 前端 warning 会展示本机预搜传回的原因，不再只看到 120s IPC 超时。

### 运维说明

- 正常用户无需配置，升级到 v0.2.19 后生效。
- 如果要临时恢复旧的主控端 fallback 行为，可设置环境变量：

```text
ERP_CLIENT_IMAGE_SERVER_FALLBACK=1
```

一般不建议开启，除非需要排查主控端图搜链路。

## 升级方式

桌面客户端：「设置 -> 应用更新 -> 检查更新」，下载完成后重启客户端。
