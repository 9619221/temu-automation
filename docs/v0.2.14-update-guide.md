# v0.2.14 更新说明

## 重点改动：workbench body 8MB+ 瘦身 + source_1688_image 直接带回 workbench

### 0.2.13 仍有的问题
- 「以图搜款」点完弹「IPC erp:purchase:workbench 调用超时 (120s)」
- 主控端日志看 SQL 145ms 就出结果，但 bodyLen 高达 **9.5MB**
- 客户端拿到这堆 JSON 后，Electron IPC 主进程→渲染进程的结构化克隆 + 渲染解析超过 120s，IPC 软超时触发

### 实测拆解出的根因
跑了一遍数据库统计：

```
total candidates: 585
source_payload_json total: 8466758 bytes  ← 9MB workbench 几乎全是这个
external_sku_options_json total: 54007
external_price_ranges_json total: 1170
inquiry_result_json total: 19594
```

每个候选商品的 `source_payload_json` 平均 14-21KB，里头 99% 是 1688 web mtop 的 raw 字段（aliTalk / brand / commonPositionLabels / company / marketOfferTag / trackInfoModel / tradeService / midComplexLabels …）—— 全是给内部审计/兜底准备的，**UI 一个字段都不读**。

UI 只用 `originTitle` / `aiTitle` / `originImageUrl` / `aiImageUrl` / `detailUrl` / `soldOut` / 营销监控配置等顶层字段，全都在 sourcePayload 顶层不在 raw 里。

加上 `source_1688_image` 这个 action 自己有个历史遗留 bug：只在 `payload.refreshWorkbench === true` 才回 workbench，客户端不发这个字段所以永远拿到 null，**被迫去发第二次 `/api/purchase/workbench`**——那次才是撞超时的那次。

### 0.2.14 修复
1. **`slimSourcePayloadForUi`**：所有走 workbench / sku_1688_sources 的出参，都从 `sourcePayload` 里剥掉 `raw` / `skuInfos` / `rawAlphaShopResponse` / `rawResponse` / `apiResponse` / `mockResponse` 等只对内部审计有用的重字段。DB 仍然完整保存，重新走兜底逻辑时还是能 `parseJsonObject(row.source_payload_json)` 拿到。
2. **`source_1688_image` action 默认带 workbench**：跟 `source_1688_keyword` / `preview_1688_url_specs` 等保持一致，走统一的 `getPurchaseWorkbenchForAction(payload, actor)`，客户端不再需要发第二次 workbench。
3. **`insert1688SourcingCandidate` 返回值瘦身**：action 响应里 candidates 字段同样剥掉 raw，避免 10 个候选 200KB+。

### 效果（实测对比）
- `[purchase/workbench] sql=140ms bodyLen=9511676` → 修后预期 **<1.5MB**
- 客户端不再需要二次 workbench 调用，UI 刷新延迟从 11-21s 降到 <1s
