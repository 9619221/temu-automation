# v0.2.7 更新说明

## 重点改动：1688 推单恢复（核心修复）

### 背景
0.2.6 及之前版本，**新供应商商品的 1688 推单全部失败**，错误信息：
> 商品[XXXXXX]不属于同一卖家或者规格[YYYYYY] 不属于商品[XXXXXX]

### 根因
1688 真实下单接口 `alibaba.trade.fastCreateOrder` 校验的是 **`cargoSpecId`**（货品规格 ID），跟商品页前台展示的 `skuId` 是两个独立 ID。而系统之前拿规格的三条路径在新供应商场景下全部失效：

- 遨虾 `productDetailQuery`：返回的 SKU 数据**只含 `skuId`，没有 `cargoSpecId`**（产品定位决定）
- 1688 `alibaba.product.simple.get`：仅限**老买卖关系**商家（订过单的）
- 主控端裸 fetch `detail.1688.com/offer/XXX.html`：**被 1688 反爬挡死**返 captcha 页

结果上一版只能把 `skuId` 当 `specId` 兜底推到 1688，被拒。

### 修复
利用项目里已有的 1688 持久化登录态浏览器（`automation/aliexpress-1688-cost.mjs`），加一个客户端「真 specId 抓取」中间层：

- worker 新增 action `extract_1688_skus`：登录态打开 1688 商品页，从 HTML 正则抽 `specId`/`skuId`/`specAttrs` 三元组
- ERP IPC client 模式 `preview_1688_url_specs` 优先走 worker，把真值组装成「解析规格」对话框需要的格式直接返回前端
- 修 `upsert_sku_1688_source` 主键 UNIQUE 冲突：specId/skuId/offerId 变化时不再复用旧 id

### 客户端代码加固（同一版本一并发布）
- 4 处 `/api/purchase/action` 显式 `timeoutMs: 120000`，消除 client 模式默认 8 秒「假超时」
- `build1688OrderCargoParamList` 加护栏：`mapping.external_sku_id == specId` 时主动抛错，不再让伪 specId 走到 1688 接口
- `alphaShopMcpClient` 解析严格分离 `skuId`/`specId` 字段集，去掉 `skuId/skuID/sku_id` 在 specId 兜底链里的位置（这些不是真 cargoSpecId）
- 「解析规格」对话框选规格时若 specId 缺失或与 skuId 同值，直接拒绝写入并给出修复指引

## 升级后必须做一次的操作（重要）

升级到 0.2.7 后，**首次推 1688 单之前需要给客户端的 1688 浏览器登录一次**：

1. 进入 ERP 应用 → 「**价格审核**」页（左侧或顶部导航）
2. 点页面右上角 / 工具栏的「**1688 登录**」按钮
3. 用手机 1688 App 扫码登录
4. 关掉登录窗口

之后客户端会用持久化 profile 保留登录态，所有「解析规格」/ 推单链路自动绕过 1688 反爬。

> 没做这一步直接「解析规格」会拿不到真 specId，回退到旧路径，推单仍会失败。

## 升级方式

桌面客户端：「设置 → 应用更新 → 检查更新」点一下，下载新版自动安装。

## 服务器侧

**主控端 `erp.temu.chat` 服务器代码不需要变动**。本次修复全部在客户端实现：客户端用 worker 抓到真 specId 后，把真值通过既有的 `upsert_sku_1688_source` action 写入主控端的 mapping，主控端推单时按既有逻辑读 mapping 调 1688 fastCreateOrder——不依赖主控端任何升级。
