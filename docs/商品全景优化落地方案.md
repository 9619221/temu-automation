# 商品全景（PipelineTab）优化落地方案

> 状态：方案待评审，**暂未写码**。
> 范围：先做第一层（数据底座）+ 第二层（卡片体检化）；第三层（ID 映射 + 操作闭环）另立项。
> 核心定位（用户拍板）：**补货/库存运营 · 流量/限流诊断 · 活动/机会挖掘**。前半段上品管控不优先。

---

## 一、现状诊断

「商品全景」前端 = `src/components/PipelineTab.tsx`，后端 = `electron/erp/services/multiStoreReport.cjs` 的 `buildPipelineOverview`，路由 `/api/erp/reports/pipeline-overview`。

三个硬伤：

### 1. 前半段阶段判定是「假」的
流水线画了 9 段（选品→上品中→核价→已建品→采购中→入库中→有库存→在售→需补货），但 `buildPipelineOverview` 真正能判定的只有后 5 段（销量/库存/采购单/入库单）。前半段靠云栖选品库凑（只有 `goods_id`，与 ERP 货号对不上），而 Temu 后台真实状态（待寄样/价格申报中/待创建首单/已发布）就在官方生命周期数据 `product_lifecycle` 里——总览 Tab 已经在用 `api.listRecords("product_lifecycle")`，商品全景却没接。结果「已建品（created）」成了兜底垃圾桶。

### 2. 卡片信息单薄
现在每张卡只显示 销量 / 库存 / 评分 / 退货 4 项。而商品面板物化表 `erp_report_cache`（「商品」Tab 数据源，`getProductPanelFast` 读它）里这些**全都算好了**，商品全景一个没用：曝光/点击/转化、限流、可报活动数+活动参考价、品质分、缺货件数、发货在途、热品标签、合规状态。

### 3. 实时聚合，冷态卡
`buildPipelineOverview` 现算 7 个查询 + 遍历两万多 SKU（仅 5 分钟内存缓存）。而「商品」Tab 早改成读物化表 + cron 预热（9s→9ms，踩坑总结）。商品全景还在用老办法。

## 二、核心设计决策

### 决策 A：数据底座复用「商品面板物化表」，不再自己聚合
`buildPipelineOverview` 改为读 `erp_report_cache`（`getProductPanelFast` 同源），叠加 `product_lifecycle` 做阶段判定。收益：
- 字段全（流量/活动/限流/品质/在途全有，白捡）；
- 性能好（读物化表，毫秒级，冷态不卡）；
- 与「商品」Tab 数据一致（同一张表，口径不打架）；
- 复用现有 cron（商品面板的 `refresh-product-panel.cjs` 每 10min 刷新），**无需新 migration、无需新 cron**。

### 决策 B：卡片维度从 SKU 改为 SPU（商品级）
现状商品全景是 SKU 级（`erp_skus` 每行一卡，截图里两万多张）。商品面板物化表是 SPU 级（`product_id` 一行 + `skus_detail` 子行）。改用物化表后，卡片自然变成**商品级**（一个商品一张卡，多尺码作为子行展开）。
- 利：更符合运营视角（管的是「商品」不是「尺码」），卡片数量大降，页面更清爽；
- 注意：这是交互层面的可见变化，多尺码差异（如某尺码缺货）通过卡内「需补货 N 个 SKU」标识 + 展开子行体现。

### 决策 C：阶段判定新规则（SPU 级，取最紧急态）
一个 SPU 下多 SKU 可能处于不同状态，**整卡取最该处理的阶段**，判定优先级从高到低：

| 阶段 | 判定条件（SPU 级聚合） | 数据来源 |
|---|---|---|
| `needs_restock` 需补货 | 有 30 日销量 且（建议补货 `advice>0` 或 缺货件数 `lack_qty>0`） | 物化表 |
| `selling` 在售 | 30 日销量 >0 或 生命周期=`12`(已发布到站点) | 物化表 + lifecycle |
| `in_stock` 有库存 | 本地库存>0 但无销量 | 物化表 stock |
| `inbound` 入库中 | 入库单非终态 | `erp_inbound_*`（保留现逻辑） |
| `purchasing` 采购中 | 采购单非终态 | `erp_purchase_*`（保留现逻辑） |
| `created` 已建品 | 生命周期=`10/11`(待创建/已创建首单) | lifecycle |
| `pricing` 核价 | 生命周期=`7/9`(价格申报中) | lifecycle |
| `listing` 上品中 | 生命周期=`1/3/14/15`(未发布/待寄样) | lifecycle |
| `selected` 选品 | 云栖选品库 status∈want/sourcing/sourced | yunqi（保留，不重点投入） |

> 生命周期码→中文已在 `src/utils/temuSelectStatus.ts`，键为 `product_skc_id`。

## 三、第一层落地：数据底座（后端）

### 3.1 新增/改造函数
在 `multiStoreReport.cjs` 新增 `buildPipelineOverviewFast(db, options)`：

1. 读物化表：`getProductPanelFast(db, options)` 拿到 SPU 行数组（含全部字段）；
2. 读生命周期：`SELECT mall_id, product_skc_id, status FROM erp_temu_openapi_records WHERE source='product_lifecycle'`，建 `mall_id|skc → status` Map；
3. 读采购/入库非终态 SKU 集合（沿用现有两段 SQL）；
4. 按决策 C 给每个 SPU 定阶段，组装成前端结构；
5. 按阶段 + 风险计 `summary` 计数。

**开关灰度**：`buildPipelineOverview` 入口加 `if (process.env.PIPELINE_USE_PANEL !== '0') return buildPipelineOverviewFast(...)`，旧逻辑保留兜底，`PIPELINE_USE_PANEL=0` 秒回滚。

## 四、第二层落地：体检卡（前端）

`ProductCard` 从「4 指标」升级为「商品体检卡」：按阶段/异常**只显示该商品当前最该关注的区块**，避免信息过载。三块主区块（对应用户定位）：

### 4.1 补货/库存区块
- 显示：销量 今/7/30 · 库存（可用 stock / 占用 occupy / 在途 shipping）· 建议补货 `+advice` · 缺货件数 `lack_qty`；
- 红标触发：`lackQty>0` 或 `adviceQty>0`；
- 操作按钮：「去备货」→ `/auto-purchase`。

### 4.2 流量/限流区块
- 显示：曝光 `expose` / 点击 `click` / 转化率 `conv` · 流量增长状态 `grow`；
- 红标触发：`limited=true`（高价限流）或 `grow` 为下滑；
- 限流时额外显示 当前价 vs 目标价 价差（来自限流清单口径，可后续接 `buildHighPriceFlowList`）；
- 操作按钮：限流时「去诊断」→ `/operations-workbench` 高价限流 Tab。

### 4.3 活动/机会区块
- 显示：可报活动数 `act_cnt` · 活动参考价价差（`declared_price` − `min_price`）· 热品标签 `hot_tag`；
- 高亮触发：`actCnt>0` 或 `hotTag`；
- 操作按钮：「去报活动」→ 活动报名 Tab。

### 4.4 风险辅助区（小字标签）
品质分 `score` 偏低 / 差评多 / 退货率高 / 合规缺失 → 复用现有 `RISK_LABELS` 红橙标签。

### 4.5 抽检/质检区块 —— ✅ 现成可接（官方 API）
- 数据源：`buildOpenapiQc`（`erp_temu_openapi_qc`，mig074），返回行带 `spu_id`/`product_skc_id`/`ext_code`，按 `spu_id` 对到商品全景 `product_id`；
- 显示：最近质检结果（合格/不合格）· 不合格数 `defective_qty` / 期望数 `expect_qty` · 疵点摘要 `flaw_summary` · 疵点图 `flaw_thumb`；
- 红标触发：`qc_result=2`（不合格）；
- 现实：默认只采不合格（`onlyBad`），覆盖有限（实测仅个别店有数据，多为合规标签缺失），**无数据走空态**。

### 4.6 违规/合规区块 —— 🟡 一半现成、一半缺数据源
- **现成**：合规状态 `compliance`（商品面板物化表已有，来自 SKC 合规快照）→ 显示「合规 / 待整改 / 缺合规标签」红橙标；
- **缺口**：真正的「违规 / 处罚 / 申诉」目前**无可靠商品级数据源**——`temu_operation_risk_snapshot` 仅消费了 `high_price_flow`，违规/处罚类未采集未解析；
- 处理：体检卡先接 `compliance`；违规/处罚列入「待补数据源」（见六），需先做违规中心采集+解析后再接。

### 4.7 广告区块 —— 🟡 商品级框架就绪、参数未标定（目前无数据）
- **商品级**：`ad_report_goods`（mig064，按 `product_id` 一行，能对到商品），但采集接口 `bg.glo.searchrec.ad.reports.goods.query` **参数仍在标定中**（代码默认 `skipAdGoods` 跳过），目前基本无数据；
- **店铺级**：`ad_report_mall` 有数据（曝光/点击/转化/花费/ROAS）但为店铺维度，非商品级；
- 处理：先把 goods.query 参数标定通（前置任务），通了再接商品级广告（曝光/点击/花费/ROAS）；标定前卡片弱显示「所属店广告概况」（`ad_report_mall`）或留空。

### 4.8 前端改动
- `UnifiedItem` 结构扩展上述字段（含 qc / compliance / ad）；
- `STAGES` / 阶段判定与后端对齐（补 pricing/listing 的 lifecycle 语义）；
- `ProductCard` 重构为按阶段渲染区块；保留卡片/表格双视图；
- 表格视图（`ProductTable`）增列：曝光、转化、限流、可报活动、缺货件数、质检不合格、合规状态。
- 改动文件：`src/components/PipelineTab.tsx`（唯一）。

## 五、数据覆盖现实与空态处理（重要）

流量/活动/限流/品质这几类来自 **cloud 抓包**，是被动采集——**只有运营实际逛过对应页面的店才有数据**（记忆里多次确认：品质看板仅 2/35 店有数据，评价源头也极少）。因此：

- 卡片对「无流量数据」必须显示空态（`—` 或「暂无流量数据」），**不能显示 0 误导**（0 曝光 ≠ 没采集）；
- 销量/库存/建议补货/缺货件数走**官方 API**（salesv2 全店覆盖），可信，是主力指标；
- 流量/活动/限流作为**有则锦上添花**的诊断维度，不作为阶段判定硬条件；
- 长期补全靠主动采集（评价已有扩展 SW 主动采集方案，流量/活动可同思路），不在本方案范围。

## 六、不做什么（边界）

- **第三层**：选品 `goods_id` ↔ 货号 ↔ Temu `product_id/skc` 三套 ID 映射、跨系统状态机、操作回写——跨系统、工作量大，**另立项**；
- 前半段（选品/上品/核价）**不重点投入**：仅用 lifecycle 码做只读展示，不做自动流转、不做云栖↔官方双向同步；
- 不新建统一 lifecycle 表（避免与现有多源数据冲突，维护成本高）。

### 待补数据源（接体检卡前需各自单独排期）
| 数据源 | 现状 | 前置任务 |
|---|---|---|
| 违规/处罚/申诉 | `temu_operation_risk_snapshot` 仅采了 `high_price_flow`，违规处罚类未采集未解析 | 做违规中心页采集 + cloud parser 解析 + 物化（参考品质看板/评价采集模式） |
| 商品级广告 | `ad_report_goods` 表结构就绪，但 `bg.glo.searchrec.ad.reports.goods.query` 参数未标定、默认 skip | 标定 goods.query 接口参数 → 去掉 `skipAdGoods` → cron 采集铺底 |

## 八、实施路线图

### 阶段 1 · 现成能接（✅ 已实现，本地未提交未发版，tsc 0 错）
数据底座换成「官方版库存销量（全店权威）主体 + cloud 流量/活动/限流/合规增强 join」，体检卡接入：补货/库存、流量/限流、活动/机会、风险、**抽检（官方质检）**、**合规状态**。
- 后端：`multiStoreReport.cjs` 新增 `buildPipelineOverviewFast`（主体 `_buildProductPanelOfficialFresh` + cloud 4 段增强 + lifecycle/qc join + 采购/入库集合 + SPU 阶段判定）+ 入口开关 `PIPELINE_USE_PANEL`；
- 路由：`lanServer.cjs`、`ipcHandlers/reports.cjs` 的 pipeline-overview 传 `attachCloudDb`；
- 前端：`PipelineTab.tsx` 体检卡化 + 表格增列；
- 验收：tsc 0 错、性能毫秒级、阶段判定准、空态正确、开关可回退。

### 阶段 2 · 违规中心采集（前置任务，单独排期）
目标：让「违规/处罚/申诉」有商品级数据源。
- 扩展/worker 抓违规中心页 → cloud parser 解析 → 物化表（参考品质看板 `buildQualityPanel`、评价采集模式）；
- 完成后：体检卡违规区块从「仅合规状态」升级为「合规 + 违规/处罚/申诉」。

### 阶段 3 · 商品级广告参数标定（前置任务，单独排期）
目标：让 `ad_report_goods` 真正有数据。
- 标定 `bg.glo.searchrec.ad.reports.goods.query` 接口参数（注释：参数较 mall 维度严格）→ 去掉 `skipAdGoods` → 开 cron 采集铺底；
- 完成后：体检卡广告区块接入商品级 曝光/点击/花费/ROAS（标定前先用 `ad_report_mall` 做「所属店广告概况」弱提示）。

> 阶段 2、3 互相独立，可并行或按优先级排；都不阻塞阶段 1 上线。
