## 现状（阶段1 后）

- 主文件 `src/pages/OperationsWorkbench.tsx` 1633 行：数据层已是 SWR hooks（见 `src/hooks/useOpsReports.ts`），字典走 `useStoreScope`，全局 owner 过滤走 `src/stores/opsWorkbenchStore.ts`。
- 12 个 Tab：`overview`(总览) `pipeline`(商品全景,已是独立 PipelineTab) `todo`(今日待办) `store`(店铺,含 ad/trend 子段) `product`(商品,含 panel/diag/restock 子段) `qc`(平台质检) `quality`(商品品质) `hpf`(高价限流) `review`(评价) `stock`(备货在途) `risk`(风险待办) `activity`(活动报名)。
- 各 Tab 的 `children` JSX 重度依赖主文件作用域的共享物（下节）。

## 拆分必须按依赖顺序：先共享层，再逐 Tab

### 第 1 步：抽共享 utils → `src/utils/opsWorkbench.tsx`
把主文件**模块级纯函数/常量**（OperationsWorkbench 函数体外，约 15–251 行）移过去并 export，主文件改 import：
- 常量：`SELLER_BASE` `RESTOCK_LABELS` `LEVEL_COLOR` `TAG_COLOR` `RISK_TYPE_LABEL` `SEV_COLOR` `SEV_TEXT` `SEV_RANK` `KIND_LABEL` `ACTIVITY_TYPE_LABEL` `TODO_TYPE_TAG` `TODO_LEVEL_TEXT` `QUALITY_SITE_LABEL` `TREND_COLORS` `RESTOCK_FAST_QTY/DAYS_*` `SLOW_MOVING_*`
- 函数：`processTarget` `diagnose` `fmtNum` `fmtMoney` `fmtReviewTime` `calcAdvice` `sellThroughDays` `isSlowMoving` `NoSearchSelect`
- 组件体内但纯（不依赖 state）的：`stackCell`(990) `skusOf`(1007) `LC_SHORT`(963) 也移出；`adviceOf`(1009) 依赖 `nowHour` → 改成 `adviceOf(r, nowHour)` 显式传参。
- 注意：含 JSX 的（stackCell/NoSearchSelect）放 `.tsx`。

### 第 2 步：共享筛选栏 → `src/components/ops/OpsCommonFilters.tsx`
`commonFilters`(1173) 用 `storeFilter`/`setStoreFilter`/`storeOptions`/`searchInput`/`setSearchInput`，10 个 Tab 共用。抽成组件，props：`{ storeFilter, onStoreChange, storeOptions, searchInput, onSearchChange, extra? }`。

### 第 3 步：共享 filter state 归属
`storeFilter`/`search`/`searchInput` 是跨 Tab 单值共享（`useSessionState` 单 key）。**保留在主文件 `OperationsWorkbench`**（作为容器持有），通过 props 传给各 Tab 组件——不要下沉到各 Tab（会破坏「切 Tab 保持筛选」语义）。`ownerFilter` 已在 zustand。各 Tab 专属 filter（如 review 的 `scoreFilter`/`regionFilter`、risk 的 `sevFilter`、activity 的 `kindFilter`）随对应 Tab 组件下沉（继续用 `useSessionState`）。

### 第 4 步：逐 Tab 容器化（建议顺序：自包含 → 复杂）

每个 Tab 抽成 `src/components/ops/XxxTab.tsx`，内部：自己调对应 SWR hook 取数 + `useStoreScope()` 拿 inScope + 自持专属 filter + 自建自己的 memo/columns + 渲染。共享 filter（storeFilter/search）+ 必要 handler 经 props 传入。

推荐顺序（依赖由少到多）：
1. **review 评价**（最自包含）：`reviewView`(1239)+`reviewStats`(1253)+`reviewColumns`(1064)+items(1538)；props: `{ storeFilter, search }`；自持 `scoreFilter`/`regionFilter`；用 `useReviews`。
2. **qc 平台质检**：`qcView`+`qcColumns`(1010)+items；用 `useOpenapiQc`+`openFlawImages`(需留主文件或一并移)。
3. **hpf 高价限流**：`hpfView`+`hpfColumns`(1152)+items；用 `useHighPriceFlow`。
4. **quality 商品品质**：`qualityView`+`qualityColumns`(1090)+`qualityShops`+items；用 `useQualityPanel`。
5. **stock 备货在途**：`stockView`+items；用 `useStockOrders`。
6. **risk 风险待办**：`riskView`(916)+`riskOverview`(911)+`riskColumns`+items；用 `useRiskList`。
7. **store 店铺**：`adAgg`/`adView`/`overviewTrend`/`trendRows` + ad/trend 子段（`storeSeg`）；用 `useAdReport`+`useSalesTrend`。
8. **todo 今日待办**：`todoTasks`(831)+`todoCount`+`todoView`+`todoColumns`(856)；依赖 diagnosed+riskRows+actRows+`markTask`+`goProcess`（props 传 handler）。
9. **product 商品**：最复杂，含 panel/diag/restock 三子段（`prodSeg`），`panelView`/`diagView`/`restockView`/`panelColumns`(1102)/`diagColumns`(831)；用 `useProductPanel`+`useSkuSales`+`useLifecycle`。
10. **activity 活动报名**：含 `submitEnroll`/`submitViaExtension`/`enrollDraft`/报名弹窗，最耦合；用 `useActivityList`。
11. **overview 总览**：聚合多源（`storeMatrix`(962) 依赖 8 个 rows + `shopAgg`/`riskOverview`/`overview`/`overviewTrend`），跨 Tab 数据最多，建议**最后**拆或保留内联。

`pipeline` 已是独立 PipelineTab，无需动。

## 关键约束
- 每个 Tab 拆完单独 `npx tsc --noEmit` + vite build 验证。
- **必须 electron 真机验证**（tsc/vite 抓不到运行时）：起 dev 连生产，逐 Tab 点开看数据正常加载、筛选/搜索/弹窗/报名提交可用。重构最易错的是 props 漏传、memo 依赖丢失、handler 闭包。
- 逐 Tab 提交（每个一个 commit，可回退）。
- 业务逻辑零改动，只搬位置 + 改数据来源（已是 hooks）+ 接 props。
- 全部完成后：tsc + vite build + electron 全 Tab 验证 → bump 版本 → 打包 → 推 erp+GitHub → 验证 latest.yml（桌面端 UI 文件不需同步服务器）。

## 风险
- props drilling：overview/todo/product/activity 依赖大量共享 memo/handler，props 会偏多。可考虑用 React Context 包一层共享只读派生（storeOwnerMap/inScope/共享 filter），但那是更大改动，建议先 props 跑通再说。
- 回归：无测试保护，唯一防线是 electron 逐 Tab 手测，务必每个 Tab 都点到。
