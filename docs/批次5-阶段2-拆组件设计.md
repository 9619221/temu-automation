# 批次5 阶段2：运营工作台拆 Tab 组件 — 设计文档

> 阶段1（SWR 数据层 + zustand + 字典共享）已完成并提交（commit `0d5678d`）。
> 本文档是阶段2「把 `OperationsWorkbench.tsx`(1633 行) 的 12 个 Tab 拆成独立组件」的完整设计，供后续专注会话照做。
> **性质提醒**：纯文件组织重构，零性能收益（前端渲染已 <33ms），高回归风险，无自动化测试保护。完成后需重新走桌面端发版流程。

## 现状（阶段1 后）

- 主文件 `src/pages/OperationsWorkbench.tsx` 1633 行：数据层已是 SWR hooks（见 `src/hooks/useOpsReports.ts`），字典走 `useStoreScope`，全局 owner 过滤走 `src/stores/opsWorkbenchStore.ts`。
- 12 个 Tab：`overview`(总览) `pipeline`(商品全景,已是独立 PipelineTab) `todo`(今日待办) `store`(店铺,含 ad/trend 子段) `product`(商品,含 panel/diag/restock 子段) `qc`(平台质检) `quality`(商品品质) `hpf`(高价限流) `review`(评价) `stock`(备货在途) `risk`(风险待办) `activity`(活动报名)。
- 各 Tab 的 `children` JSX 重度依赖主文件作用域的共享物（下节）。

## 拆分必须按依赖顺序：先共享层，再逐 Tab

### 第 1 步：抽共享 utils → `src/utils/opsWorkbench.tsx`

> ⚠️ 行号以当前主文件（1662 行）为准，下同。本步是把一切**不依赖组件 state/props 的常量、纯函数、纯展示片段**全部移到 utils 并 export，主文件 import。**清单务必抽全——漏一个，后面拆 Tab 时就会缺引用、临时回头补，破坏「逐 Tab 干净 commit」的节奏。**

**(A) 模块级纯常量**（函数体外，125–253 行附近）：
`SELLER_BASE` `RESTOCK_LABELS` `LEVEL_COLOR` `TAG_COLOR` `RISK_TYPE_LABEL` `SEV_COLOR` `SEV_TEXT` `SEV_RANK` `KIND_LABEL` `ACTIVITY_TYPE_LABEL` `TODO_TYPE_TAG`(164) `TODO_LEVEL_TEXT`(167) `QUALITY_SITE_LABEL`(126) `TREND_COLORS`(227) `RESTOCK_FAST_QTY/RESTOCK_DAYS_*`(231-233) `SLOW_MOVING_DAYS/SLOW_MOVING_ONLINE_DAYS`(242-243)

**(B) 模块级纯函数（零 JSX）**（函数体外）：
`processTarget`(173) `diagnose`(196) `fmtNum`(216) `fmtMoney`(217) `fmtReviewTime`(219) `calcAdvice`(234) `sellThroughDays`(244) `isSlowMoving`(252)

> 第 1 步只抽 (A)+(B)——纯数据常量 + 纯逻辑函数，**不含任何 JSX**。类型从 `../types/opsWorkbench` import（`SkuRow`/`Diag`/`TodoTask`/`ProductPanelRow`/`SkuChild`，已存在）。这部分零类型风险、`tsc` 必过，是干净的起点。文件命名 `opsWorkbench.tsx`（为 (C) 组后续迁入留位），第 1 步内容暂无 JSX 也合法。

**(C) 组件体内的纯片段——第 1 步暂不动，留到拆对应 Tab 时随该 Tab 迁移**（原因：多是单 Tab 专属，且 `storeCol`/`skuTitleCol` 抽成独立常量会触发 antd 列类型协变检查，徒增风险）：
- `NoSearchSelect`(18 模块级，但含 JSX) — 所有 Tab 的分页器用；可在第 1 步之后单独一步抽到 utils（要 `import React`/`Select`），也可暂留主文件。
- `storeCol`(843) `skuTitleCol`(830) `stackCell`(990) `estTextW`(986) `skusOf`(1007) `adviceOf`(1009) → 随 **ProductTab** 迁（diag/restock/panel 三子段都在 product）。`adviceOf` 依赖 `nowHour`(1008=`new Date().getHours()`)，迁移后 `nowHour` 在 ProductTab 组件体内算，**不要**抽成模块级常量（否则页面开着跨小时不更新）。
- `redNum`(962) `LC_SHORT`(963) `LIFECYCLE_STAGE_ORDER`(737) → 随 **OverviewTab** 迁（仅 storeMatrix/storeMatrixColumns 用）。
- `SRC_LABEL`(948) → 随 **StockTab** 迁。
- 若迁移时发现某片段被两个以上**不同** Tab 真正共用，再回头提到 utils；否则就近放在使用它的 Tab 组件里。

### 第 2 步：共享筛选栏 → `src/components/ops/OpsCommonFilters.tsx`
`commonFilters`(1173) 用 `storeFilter`/`setStoreFilter`/`storeOptions`/`searchInput`/`setSearchInput`，10 个 Tab 共用。抽成组件，props：`{ storeFilter, onStoreChange, storeOptions, searchInput, onSearchChange, extra? }`。

> 注意输入框绑的是 `searchInput`（跟手），下游过滤用的是防抖后的 `search`——见第 3 步，这是两个值，别混。`storeOptions`(581) 派生自 `skuRows`，由容器算好传入（见下）。

### 第 3 步：共享 filter state 归属
- `storeFilter`/`searchInput` 是跨 Tab 单值共享（`useSessionState` 单 key）。**保留在主文件 `OperationsWorkbench`** 作为容器持有，props 传给各 Tab——不要下沉（会破坏「切 Tab 保持筛选」语义）。
- `search = useDebouncedValue(searchInput, 250)`(296) 的**防抖计算也留在容器**，把算好的 `search`（防抖值）一并 props 传给各 Tab 做过滤；各 Tab **不要**自己再 import `useDebouncedValue` 重算（会各 Tab 各一份防抖时序，且与筛选栏绑定的 `searchInput` 脱节）。
- `storeOptions`(581) 派生自 `skuRows`，留容器算，props 传给 `OpsCommonFilters`。
- `ownerFilter`/`inScope` 已在 zustand + `useStoreScope`，各 Tab 自己 `useStoreScope()` 取，不走 props。
- 各 Tab 专属 filter（review 的 `scoreFilter`/`regionFilter`/`qualitySiteFilter`、risk 的 `sevFilter`、activity 的 `kindFilter`、product 的 `prodSeg`/`diagFilter`/`slowFilter`/`onsaleDaysFilter`、store 的 `storeSeg`、todo 的 `todoType`/`todoStatus`）随对应 Tab 组件下沉（继续用 `useSessionState`，key 不变以保留持久化）。

### 第 3.5 步：容器固定职责——两个全局弹窗 + 预览，别下沉

主文件 return 里有两个 Modal 和一个隐藏预览，是**跨 Tab 共享的容器级 UI**，拆 Tab 时**留在容器**，由相关 Tab 经 props 接触发器。这是文档原版最大的隐患（product 和 hpf 隔着 6 个 Tab 拆，极易漏接趋势弹窗）：

1. **销量趋势弹窗**：`trendOf`/`trendModalRows`/`trendModalLoading` 三个 state + 拉数 `useEffect`(542) + 末尾 Modal(1645) 全部留容器。触发器 `setTrendOf` 被 **product(1113) 和 hpf(1160) 两个 Tab** 调用 → 两个 Tab 都要 props 接 `onOpenTrend(productId, title)`。
2. **疵点照片预览**：`flawPreviewVisible`/`flawPreviewImages` 两个 state + `openFlawImages`(530) 回调 + 隐藏的 `Image.PreviewGroup`(1622) 留容器。只 **qc** 用 → QcTab props 接 `onOpenFlawImages(mallId, qcBillId)`。
3. **活动报名弹窗**（`enrollModalSku` + `submitEnroll`/`submitViaExtension`/`enrollDraft`/批量填值等）只 activity 用，且高度自包含 → 整套随 ActivityTab 下沉，**不**留容器（与上面两个不同，注意区分）。

### 第 4 步：逐 Tab 容器化（建议顺序：自包含 → 复杂）

每个 Tab 抽成 `src/components/ops/XxxTab.tsx`，内部：自己调对应 SWR hook 取数 + `useStoreScope()` 拿 inScope + 自持专属 filter + 自建自己的 memo/columns + 渲染。共享 filter（storeFilter/search）+ 必要 handler 经 props 传入。

**分两批：第一批走纯 props（自包含 Tab），第二批走 Context（强耦合 Tab）。** 分批理由见「关键决策：props vs Context」。

**第一批 · 纯 props（共享 props 仅 `{ storeFilter, search }`，外加各自的弹窗触发器）**：

2. **qc 平台质检**：`qcView`(1181)+`qcColumns`(1011)+items(1473)；用 `useOpenapiQc`；**props 接 `onOpenFlawImages`**（弹窗本体留容器，见 3.5 步）。
3. **hpf 高价限流**：`hpfView`(1221)+`hpfAgg`(1230)+`hpfColumns`(1149)+items(1521)；用 `useHighPriceFlow`；**props 接 `onOpenTrend`**（销量趋势弹窗）。
4. **quality 商品品质**：`qualityView`(1192)+`qualityDist`(1205)+`qualityShopsView`(1203)+`qualityColumns`(1031)+items；自持 `qualitySiteFilter`(1191)；用 `useQualityPanel`。
5. **stock 备货在途**：`stockView`(804)+`stockColumns`(949)+items(1558)；用 `useStockOrders`。

### 关键决策：props vs Context

todo/product/overview 这三个 Tab 的 memo 横跨多个数据源、还要 `inScope`/`markTask`/`goProcess`/`jumpWorkbench` 等一堆容器级 handler。如果硬走纯 props，单个组件 props 会有十几个，难维护也易漏传。**对这三个（含 activity 的部分 handler）建一层只读 Context**：

```
src/components/ops/OpsWorkbenchContext.tsx
  提供：inScope / storeOwnerMap（其实 useStoreScope 已给，可不进 Context）
        共享 filter：storeFilter / search
        跨 Tab handler：jumpWorkbench / goProduct / markTask / goProcess / onOpenTrend
        navigate
```

第一批 7 个 Tab **不**用 Context（它们只要 `storeFilter`/`search` + 至多一个弹窗触发器，props 足够清爽）；第二批进 Context。这样既不过度抽象，又避免强耦合 Tab 的 props 爆炸。注意 Context value 要 `useMemo` 包稳定，否则 Provider 每次 render 会让所有消费 Tab 重渲染（虽前端已 <33ms，但别白白制造无效渲染）。

## 关键约束
- 每个 Tab 拆完单独 `npx tsc --noEmit` + vite build 验证。
- **必须 electron 真机验证**（tsc/vite 抓不到运行时）：起 dev 连生产，逐 Tab 点开看数据正常加载、筛选/搜索/弹窗/报名提交可用。重构最易错的是 props 漏传、memo 依赖丢失、handler 闭包。
- 逐 Tab 提交（每个一个 commit，可回退）。
- 业务逻辑零改动，只搬位置 + 改数据来源（已是 hooks）+ 接 props。
- 全部完成后：tsc + vite build + electron 全 Tab 验证 → bump 版本 → 打包 → 推 erp+GitHub → 验证 latest.yml（桌面端 UI 文件不需同步服务器）。

## 风险
- **共享物抽漏**：第 1 步清单（尤其 (C) 组）必须抽全，否则拆到一半发现缺 `storeCol`/`stackCell`/`redNum` 等又回头补，打断 commit 节奏。开工前先把第 1 步做完并 `tsc` 过，再动 Tab。
- **弹窗触发器漏接**：trend 弹窗(product/hpf)、flaw 预览(qc) 的本体留容器、触发器走 props——拆这三个 Tab 时最易忘，3.5 步已单列，逐项核对。
- **props vs Context 边界**：第一批纯 props、第二批 Context，别混；Context value 记得 `useMemo` 包稳定。
- **回归**：无测试保护，唯一防线是 electron 逐 Tab 手测。每个 Tab 必点：数据加载 / 店铺筛选 / 搜索（含防抖跟手）/ 子段切换（store、product）/ 弹窗（trend、flaw、报名）/ 报名提交（单店 + 多店扩展）/ 待办「完成·忽略·恢复」闭环 / 跨 Tab 跳转（pipeline 风险标签 → 对应 Tab）。
- **切 Tab 保持筛选**：`storeFilter`/`searchInput` 留容器是为了这个语义，拆后务必验证「在 A Tab 搜了关键词，切到 B Tab 仍保留」。
